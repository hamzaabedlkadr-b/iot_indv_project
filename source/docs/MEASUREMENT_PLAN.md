# Measurement Plan

This file defines what we need to measure for the final report.

## Required Measurements

### 1. Maximum Sampling Frequency

Measure the highest sampling frequency supported by the ESP32 implementation in two ways:

- raw class-style benchmark for maximum sample-generation throughput
- strict full-pipeline benchmark for the conservative stable operating point

Record:

- raw achieved frequency,
- configured full-pipeline frequency,
- actual achieved full-pipeline behavior,
- failure point or instability symptoms.

### 2. Adaptive Sampling Benefit

Compare:

- fixed oversampling baseline
- adaptive sampling mode

Record:

- average sampling frequency,
- number of samples per window,
- whether signal quality remains acceptable.

### 3. Aggregate Computation

For each window, record:

- window index,
- signal profile,
- anomaly count if the profile injects spikes,
- window duration,
- average value,
- dominant frequency,
- selected sampling frequency.

### 4. Execution Time

Measure the time spent in:

- sampling,
- FFT processing,
- aggregate computation,
- communication handling.

### 5. Network Volume

Measure:

- bytes sent in fixed mode,
- bytes sent in adaptive mode.

### 6. End-to-End Latency

Measure from:

- data generation on the board,
- to aggregate reception at the edge server.

## Energy Measurement Plan

Energy is a graded requirement, so we should decide the method before test day.

The concrete execution steps now live in:

- `docs/ENERGY_MEASUREMENT_RUNBOOK.md`
- `docs/INA219_ENERGY_SETUP.md`
- `firmware/ina219_power_monitor/README.md`

Preferred order:

1. direct measurement with the `INA219` two-board setup,
2. timed baseline versus adaptive runs under the same conditions,
3. saved notes about instrumentation limits if only an approximate comparison is possible.

For each energy run, record:

- measurement method,
- run duration,
- communication mode,
- average selected sampling frequency,
- total transmitted messages,
- observed power or energy value,
- any uncertainty or limitations.

For this project, the minimum meaningful graded comparison is:

- baseline fixed `50 Hz`
- adaptive run that settles to `40 Hz`

under:

- `clean_reference`
- `MQTT_ONLY`
- equal run duration

The deep-sleep examples in some reference repos are useful as additional context, but they should not replace this core comparison.
